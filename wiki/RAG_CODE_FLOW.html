一、架构对比：Phase 1 vs Phase 2

Phase 1 — 全量注入

用户输入
↓ tag 关键词匹配
↓ 拼接全部 25 条 FAQ → Prompt
↓ DeepSeek LLM（单次调用）
↓ 回答
缺陷：FAQ 越多 Prompt 越长，token 浪费严重，LLM 容易答非所问。

Phase 2 — RAG 检索增强 NEW

用户输入
↓ Embedding 模型向量化
↓ Milvus 向量数据库检索 Top-3
↓ 取最相关的 3 条 FAQ → Prompt
↓ DeepSeek LLM（单次调用）
↓ 回答
优势：Prompt 极短，只注入相关内容，召回精准，token 消耗大幅降低。
💡 关键区别一句话总结
Phase 1 把所有 FAQ 都塞进 Prompt，LLM 自己大海捞针； Phase 2 先用向量检索把最相关的 3 条找出来，再送给 LLM，LLM 只需要处理很少的数据。

三、逐步骤详解

① HTTP 请求进入 FastAPI Phase 2 ▶

用户在前端页面发送消息，前端以 POST /api/chat 请求体发送 JSON：

// 请求体（app/api/chat.py）
class ChatRequest(BaseModel):
    message: str                // 用户输入，如"网络断了怎么办"
    history: Optional[List[ChatMessage]]  // 前几轮对话
    session_id: Optional[str]  // 会话标识（可无）

FastAPI 路由层接收请求，验证数据合法性，然后交给 chat() 函数处理。这一步是 HTTP 层，和 Phase 1 完全相同。

② FastAPI chat() — 计时 + session_id 生成 Phase 2 ▶

记录请求开始时间，用于计算响应延迟：

def chat(request: ChatRequest):
    start_time = time.time()               // 记录开始时间
    session_id = request.session_id or f"session_{int(start_time * 1000)}"  // 无则生成

如果前端没有传 session_id，则用时间戳自动生成一个。这保证了客户端能跟踪同一会话。

⚡ 优化点 1：History 每次重建，效率低
Phase 2 中前端每次请求都需要把完整的对话历史带过来。如果前端没有维护历史（只传最新一条），多轮对话上下文会丢失。 建议在服务端用 Redis/内存维护会话历史，避免前端重复传历史数据，也能减少网络传输量。
③ format_chat_history() — 历史对话格式化 Phase 2 ▶

将前端传来的对话历史列表格式化为纯文本字符串，注入 Prompt 的 {history} 占位符：

// app/llm/deepseek_client.py
def format_chat_history(messages):
    # 输入: [{"role": "user", "content": "怎么创建会议"},
    #        {"role": "assistant", "content": "点击新建..."}]
    # 输出:
    #   用户: 怎么创建会议
    #   助手: 点击新建...
    for msg in messages:
        role = msg.get("role", "user")
        if role == "user":   formatted.append(f"用户: {content}")
        elif role == "assistant": formatted.append(f"助手: {content}")

这只是格式转换，没有调用外部服务，是纯内存操作，速度极快。

⚡ 优化点 2：历史对话无压缩，会无限增长
随着对话轮次增加，history_text 会越来越长，最终消耗大量 token。 建议在 format_chat_history 中实现摘要式压缩：只保留最近 N 轮，或使用 LLM 将历史压缩为摘要。
④ invoke_rag_chain() — Phase 2 核心入口 Phase 2 新增 ▶

这是 Phase 2 新增的函数，是整个 RAG 流程的触发点：

// app/chains/faq_chain.py
def invoke_rag_chain(question: str, history: str = "") -> str:
    chain = build_rag_chain(history=history)   // 每次请求都重建 Chain ⚠️
    return chain.invoke(question)              // 执行 Chain，返回字符串
⚡ 优化点 3：每次请求都重建 Chain（重要）
build_rag_chain() 内部创建了 ChatOpenAI 实例、ChatPromptTemplate 对象， 这些对象可以复用，不需要每次请求都重新创建。应该在模块级别或通过缓存复用。

正确做法：LLM 实例和 Prompt 模板在模块加载时创建一次，invoke_rag_chain 只负责接收参数、调用固定 Chain。
⑤ build_rag_chain() — LCEL 管道拼装 Phase 2 核心 ▶

使用 LangChain LCEL（LangChain Expression Language）把三个组件用 | 管道符串联：

// app/chains/faq_chain.py
def build_rag_chain(history=""):
    llm = get_llm()                                        // ① 获取 DeepSeek LLM 实例
    prompt = ChatPromptTemplate.from_messages([
        ("system", SYSTEM_PROMPT_V2),                // ② 系统 Prompt 模板
        ("human", "{question}"),                   // ③ 用户问题占位符
    ])
    // ④ LCEL 管道：input dict → prompt.format() → llm.invoke() → StrOutputParser
    chain = (
        {
            "context":  lambda q: _format_context(_retrieve_docs(q)),  // 检索 + 格式化
            "question": RunnablePassthrough(),                          // 原样传递 question
            "history":  lambda _: f"【历史对话】\n{history}" if history else "【历史对话】（无）",
        }
        | prompt    // 把 dict 注入 Prompt，替换 {context}/{history}/{question}
        | llm       // LLM 调用 DeepSeek API
        | StrOutputParser()  // 提取 LLM 返回的纯文本
    )
    return chain
⚡ 优化点 4：Lambda 在每次调用时重新执行，高并发下有性能问题
管道中的 lambda q: _format_context(_retrieve_docs(q)) 在 Chain 执行时才被调用， 意味着 Milvus 检索和 DeepSeek LLM 调用是串行的。
可以将 LLM 初始化和 Prompt 模板提到模块级别（全局单例），避免每次请求重建。
⑥ _retrieve_docs() — Milvus 向量检索（Phase 2 核心新增） Phase 2 新增 ⭐ ▶

这是 Phase 2 最关键的步骤：把用户问题变成向量，然后在 Milvus 里找最相似的 3 条 FAQ。

6.1 获取 Embedding 实例

// app/llm/embedding_client.py
def get_embedding_client():
    return HuggingFaceEmbeddings(
        model_name="sentence-transformers/all-MiniLM-L6-v2",  // 模型名，来自 .env
        model_kwargs={"device": "cpu"},                           // 本地 CPU 运行，无需 GPU
        encode_kwargs={"normalize_embeddings": True},         // L2 归一化，提升相似度计算精度
    )

使用 HuggingFace 的本地 Embedding 模型，免费、离线、无需 API key。 normalize_embeddings=True 让向量长度归一化为 1，这样向量点积（IP 内积）就等于余弦相似度。

6.2 生成用户问题的向量

// _retrieve_docs 内部
vector = embeddings.embed_query("网络断了怎么办")  // → 返回 384 维浮点数数组
// 形状: [0.023, -0.041, 0.118, ..., -0.007]  (共 384 个数)

每条 FAQ 在建索引时（scripts/build_milvus_index.py）就已经被这个模型转成 384 维向量存入 Milvus。 现在用户问"网络断了怎么办"，系统也把它变成向量，然后在 150 条 FAQ 向量中找最近的 3 个。

6.3 Milvus 向量相似度检索

// _retrieve_docs 内部（app/chains/faq_chain.py）
from pymilvus import MilvusClient

client = MilvusClient(uri=f"http://{host}:{port}")
results = client.search(
    collection_name="zoo_faq_collection",   // Milvus collection 名称
    data=[vector],                           // 用户问题的向量
    limit=3,                                 // 只取 Top-3
    output_fields=["faq_id", "tags", "text"],  // 返回哪些字段
)

Milvus 使用 IVF_FLAT 索引，在 150 条向量中快速找到最相似的 3 条。返回结果包含相似度分数（distance）。

6.4 将 Milvus 返回结果转换为 LangChain Document

// _retrieve_docs 内部
docs = []
for hit in results[0]:              // results[0] 是主查询结果列表
    entity = hit.get("entity", {})   // Milvus 返回的字段
    docs.append(Document(
        page_content=entity.get("text", ""),     // 索引时的 enriched 检索文本
        metadata={
            "faq_id": entity.get("faq_id", "?"),
            "tags":    entity.get("tags", ""),
            "score":   hit.get("distance", 0.0),
        },
    ))
⚡ 优化点 5：Embedding 模型对中文支持较弱（重要）
当前使用的 all-MiniLM-L6-v2 是英文优化模型，对中文语义理解能力有限。 "网络断了怎么办" 和 "断网" 的中文词向量在这个英文模型中相似度可能偏低。

建议替换为中文 Embedding 模型，例如： shibing624/text2vec-base-chinese（中文语义效果优秀，768 维）或 BAAI/bge-large-zh-v1.5（中文 BGE，1024 维，中文效果最佳）。
⚡ 优化点 6：Milvus 连接每次新建
每次调用 _retrieve_docs 都会新建一个 MilvusClient 连接。 在高并发场景下，频繁创建销毁连接会有额外开销。建议使用连接池或全局复用单例客户端。
⚡ 优化点 7：缺少召回质量评估
当前没有任何手段知道召回的 3 条 FAQ 是否真的和用户问题相关。 建议记录每次检索的 distance 分数，如果最高分数低于阈值，触发降级逻辑（如回退到 Phase 1 全量匹配）。
⑦ _format_context() — 把检索结果格式化为 Prompt 文本 Phase 2 ▶

将 Document 列表转换为供 LLM 读取的纯文本：

// app/chains/faq_chain.py
def _format_context(docs: list[Document]) -> str:
    if not docs:
        return "（未找到相关 FAQ，请告知用户暂无相关信息）"
    lines = []
    for doc in docs:
        faq_id = doc.metadata.get("faq_id", "?")
        lines.append(f"【问题 {faq_id}】{doc.page_content}")
    return "\n".join(lines)

输入示例（docs 长度为 3）：

// 格式化后的 {context} 字符串示例
【问题 18】断网 网络断了 网络中断 连不上 掉线 ... 会议中断网了怎么办 ...
【问题 5】听不到 声音问题 ... 会议中听不到声音怎么办 ...
【问题 95】网络卡顿 网络不稳定 连接优化 ...
⚡ 优化点 8：page_content 是 enriched 检索文本，不是原始问答
建索引时存入 Milvus 的 text 是经过变体增强的混合文本（包含 tags、问法变体、关键词），LLM 看到的是增强文本而非原始清晰问答。 建议额外存储原始 question 和 answer 字段，检索时取原始字段填充 context。
⑧ ChatPromptTemplate — 系统提示词组装 Phase 2 ▶

SYSTEM_PROMPT_V2 是 Phase 2 的系统提示词模板，包含三个占位符：

// app/chains/faq_chain.py
SYSTEM_PROMPT_V2 = """你是一个专业的 Zoo 会议服务客服助手。

以下是与用户问题相关的 FAQ 参考内容：
{context}

请根据以上参考内容回答用户问题。如果找不到相关信息，请礼貌告知用户。
{history}"""

当 LCEL 管道执行时，三个占位符被替换为：

  • {context} → _format_context() 返回的 Top-3 FAQ 格式化文本
  • {history} → 历史对话格式化文本（无历史时为"【历史对话】（无）"）
  • {question} → 用户原始输入（如"网络断了怎么办"）

完整 Prompt 示例（实际发送给 DeepSeek 的内容）：

// ===== System =====
你是一个专业的 Zoo 会议服务客服助手。

以下是与用户问题相关的 FAQ 参考内容：
【问题 18】断网 网络断了 网络中断 连不上 掉线 ... 会议中断网了怎么办 ...
【问题 5】听不到 声音问题 ... 会议中听不到声音怎么办 ...
【问题 95】网络卡顿 网络不稳定 连接优化 ...

请根据以上参考内容回答用户问题。如果找不到相关信息，请礼貌告知用户。
【历史对话】
（无）

// ===== Human =====
{question}
⑨ get_llm() — 获取 DeepSeek LLM 实例 Phase 2 ▶

DeepSeek 提供 OpenAI 兼容 API，所以用 LangChain 的 ChatOpenAI 封装接入：

// app/llm/deepseek_client.py
def get_llm() -> BaseChatModel:
    return ChatOpenAI(
        model="deepseek-chat",                  // DeepSeek 模型名
        base_url="https://api.deepseek.com",  // DeepSeek API 地址
        api_key="sk-...",                      // 从 .env 读取
        temperature=0.7,                         // 创造性参数（0~1）
        max_tokens=512,                        // 最大回复 token 数
    )

配置项全部从 .env 读取，通过 config/settings.py 的 get_settings() 注入。

⚡ 优化点 9：LLM 实例未复用（高优先级）
当前每次请求都调用 get_llm() 创建新实例，应该在模块加载时创建一次全局实例并复用。 get_llm() 内部应加缓存（@lru_cache 或全局变量），避免重复创建 ChatOpenAI 对象。
⑩ DeepSeek API 调用 — LLM 生成回答 Phase 1 & Phase 2 共有 ▶

LangChain 把组装好的 Prompt 通过 HTTP 发送到 DeepSeek API，DeepSeek 返回生成的回答。

这一步 Phase 1 和 Phase 2 调用的都是同一个 DeepSeek 接口，区别仅在于发送的 Prompt 内容完全不同：

  • Phase 1 Prompt：包含全部 25 条 FAQ（~2000 token）
  • Phase 2 Prompt：只含 Top-3 相关 FAQ（~300-500 token）
⚡ 优化点 10：未接入 Streaming 响应
当前是"等待 LLM 完整生成后才返回"，用户需要等待几秒才能看到任何内容。 接入 LangChain Streaming（ChatOpenAI(streaming=True)）后可以实时逐字返回，大幅提升用户体验。
⑪ StrOutputParser — 提取纯文本 Phase 2 ▶

LLM 返回的是一个 AIMessage 对象（含 metadata），StrOutputParser() 负责提取其中的纯文本部分：

// LangChain 内部行为示意
// LLM 返回: AIMessage(content="会议中断网了...", additional_kwargs={...})
// StrOutputParser 提取: "会议中断网了，请检查..."（纯字符串）

这一步很简单但不可缺少，保证最终返回给前端的是干净的字符串。

⑫ FastAPI 返回响应 Phase 1 & Phase 2 共有 ▶

将 LLM 回答封装为 HTTP JSON 响应返回给前端：

// app/api/chat.py
elapsed_ms = (time.time() - start_time) * 1000
return ChatResponse(
    answer=answer,              // LLM 生成的回答
    session_id=session_id,      // 会话 ID（前端可用于跟踪上下文）
    latency_ms=round(elapsed_ms, 2),  // 响应耗时（毫秒）
)

响应示例：

{
  "answer": "会议中断网了，请按以下步骤操作...",
  "session_id": "session_1746598321000",
  "latency_ms": 1847.32
}
