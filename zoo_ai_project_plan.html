第 1 阶段 构建最小可运行的 FAQ 机器人

这是整个项目的 MVP（最小可行产品）。你将学会：如何用 LangChain 串联 LLM 与知识库、如何构建一个完整对话流程。

任务清单

Week 3：调用 LLM，实现最简对话

  1. 注册 OpenAI 账号，获取 API Key（或其他兼容 API，如硅基流动、DeepSeek 等国产平替）

    • Chain：把多个步骤串联成一个 pipeline
    • Memory：保存对话历史
  2. 用 LangChain 构建一个 FAQ 问答 Chain：
    • 用户提问 → PromptTemplate 构造提示词 → LLM 生成回答 → 返回结果
  3. 为 Zoo 的会议服务（Meetings）产品线，手写 20~30 条 FAQ（如"如何创建会议"、"如何共享屏幕"等）
  4. 让 LLM 只基于这些 FAQ 回答，超出范围的礼貌拒绝

Week 5：包装成 Web 服务

  1. 学习 FastAPI 基础：路由定义、请求体、响应体
  2. 用 FastAPI 包装你的 FAQ 机器人：
    • `POST /chat` 接收用户消息
    • 返回 AI 回答
  3. 前端做一个极简 HTML 页面，调用后端 API，实现实时对话
核心概念你要理解透
  • Chain = 工序流水线。每个工序（Prompt → LLM → Parser）是一个环节，多个环节串联成完整流程

第 2 阶段 引入 RAG 与向量检索

预计时间：第 6~9 周

这是 AI 客服系统的核心能力。你将学会：如何把文档知识存入向量数据库，查询时如何精准召回相关内容（Recall）。

核心技术点

Embedding 向量数据库 FAISS / Chroma LangChain RAG Chain 文档切分 (Chunking)

任务清单

Week 6：理解 Embedding 与向量检索原理

  1. 理解什么是文本 Embedding：把文字变成一串数字向量，语义相近的文字向量也相近
  2. 用 OpenAI 的 `text-embedding-ada-002`（或国产 embedding 模型如 BGE）对 Zoo 会议服务的产品手册段落生成向量
  3. 用 余弦相似度手工计算两条文本的相似度，理解向量距离的物理意义
  4. 理解：为什么不能用关键词匹配（BM25）做语义搜索？Embedding 解决了什么问题？
关键原理
  • Embedding 模型将文本映射到高维向量空间（如 1536 维）
  • 语义相似 = 向量空间中距离近（用余弦相似度或欧氏距离衡量）
  • 召回（Recall）= 从海量向量中找到与用户问题最相似的 Top K 个
  • 这一步的质量直接决定后续 LLM 能否回答准确——"garbage in, garbage out"

Week 7：搭建向量数据库

  1. 选型向量数据库（初学推荐）：
    • ChromaDB（最简单，Python 原生，适合本地学习）
    • FAISS（Facebook 开源，支持海量向量，推荐生产级入门）
  2. 准备 Zoo 知识文档：
    • 会议服务产品手册（PDF/TXT）
    • 历史工单中的常见问题与解答（脱敏后）
    • 技术故障排查文档
  3. 实现文档切分（Chunking）策略：
    • 按固定长度切（如 500 字一段）
    • 按段落切（更语义完整，推荐）
    • 重叠切分（相邻 chunk 重叠 50~100 字，减少上下文断裂）
  4. 生成每个 chunk 的向量，存入 FAISS/Chroma
初学者常犯的错误：chunk 切得太大或太小。太大会引入太多噪声（LLM 处理不过来）；太小会丢失上下文，导致回答不完整。建议从 300~600 字开始尝试，逐步调优。

Week 8：实现 RAG 检索链

  1. 构建 LangChain RAG Chain 四步走：
    • ① 问题 Embedding：用户问题 → 向量
    • ② 向量检索：在向量库中搜索 Top K 相关 chunk
    • ③ 上下文组装：将用户问题 + 召回的 chunk 组装成 Prompt
    • ④ LLM 生成：让 LLM 基于检索到的内容回答
  2. 用 LangChain 的 `RetrievalQA` Chain 或 LCEL 写法实现上述流程
  3. 添加引用来源功能：让 LLM 的回答中标注"参考了哪篇文档的第几段"，便于用户验证
  4. 对比实验：直接让 LLM 回答 vs 加了 RAG 之后回答，测试效果差异

Week 9：扩展到更多产品线

  1. 为 Zoo 的通话服务（Phone）准备第二批文档，重复 Week 7~8 的流程
  2. 学习多产品线知识库隔离：不同产品的知识存不同 collection，避免跨产品干扰召回
  3. 用 Zoo 的产品场景做端到端测试：
    • 用户问"大屏怎么投屏" → 应该召回到会议大屏知识库
    • 用户问"耳机没声音" → 应该召回到耳机知识库
召回策略核心概念（面试/面试也能用）
  • 召回（Recall）：从知识库中把所有可能相关的候选都找出来。recall 低 = 好的答案根本没被召回，后面的 LLM 再强也没用
  • 精准度（Precision）：召回的结果中，有多少是真正相关的。precision 低 = 召回了大量无关内容，LLM 被噪声干扰
  • BM25：基于关键词的传统召回算法，RAG 中作为辅助补充（embedding 召回了 Top 20，BM25 再补充 Top 5）
  • Hybrid Search：向量检索 + BM25 混合使用，兼顾语义和关键词
做完后你要能回答：RAG 的四步流程是什么？Embedding 和关键词匹配各自的优势是什么？为什么说召回是 RAG 的瓶颈？
阶段产出
基于真实文档的 RAG 机器人。能自己准备文档、切分、存入向量库，并能说出每个环节的原理。可以对比"有 RAG"和"无 RAG"回答质量的差异。
第 3 阶段 意图识别与对话管理

预计时间：第 10~13 周

你将学会：如何让 AI 理解用户到底想问什么（意图识别），如何在多轮对话中记住上下文（对话管理），以及如何设计一个能真正解决问题的对话流程。

核心技术点

意图识别 多分类模型 / LLM Classification 槽位填充 (Slot Filling) 对话状态管理 多轮追问

任务清单

Week 10：意图识别——用户到底想问什么

  1. 理解意图识别（Intent Detection）的业务价值：不同意图 → 不同处理流程
  2. 设计 Zoo 会议服务的意图分类体系（参考原文，预设 8~10 个核心意图）：
    意图 ID意图名称示例问法
    greet问候/寒暄"你好"
    meeting_create创建会议"怎么发起一个会议"
    screen_share共享屏幕"如何共享我的屏幕"
    troubleshoot_audio音频故障"对方听不到我的声音"
    schedule_meeting预约会议"我想预约明天下午3点的会"
    general_inquiry通用咨询"Zoom是什么"
    out_of_scope范围外"今天天气怎么样"
  3. 用LLM 做意图分类（最简方案）：
    • 写一个 Prompt："你是一个意图分类器。用户消息属于以下哪个类别：greet, meeting_create, screen_share, troubleshoot_audio... 请只输出类别名称。"
    • 评估准确率，观察哪些意图容易混淆
  4. 如果想进阶：了解 BERT/RoBERTa 微调做意图分类的流程（初学可跳过，后续有余力再做）
意图识别方案对比
  • LLM 分类（本阶段用）：最简单，无需训练数据，依赖 Prompt 质量。适合意图类型少、表述清晰的场景
  • 微调小模型（进阶）：用标注数据 fine-tune BERT，适合意图多、表述差异大的生产场景
  • 规则兜底：正则匹配关键词，如"会议"+"创建" → meeting_create。响应快，但覆盖率有限

Week 11：多轮对话与槽位填充

  1. 理解"槽位填充（Slot Filling）"：收集解决用户问题所需的必要信息
    • 例如：故障排查"耳机没声音"需要知道：设备型号、连接方式（Wi-Fi/有线）、发生场景（仅会议中/一直无声）
  2. 用 LangChain 的 ConversationBufferMemory 或 ConversationSummaryMemory 存储对话历史
  3. 设计对话流程图：
    • 用户："耳机没声音"
    • 机器人："请问您用的是哪款耳机型号？"
    • 用户："Poly CCX 500"
    • 机器人："请问是仅在会议中无声，还是一直都没有声音？"
    • ... 收集完槽位后，再走 RAG 检索精确答案
  4. 用 LangChain 的 Agent 机制处理条件分支（if intent == troubleshoot_audio → 进入故障排查分支）
对话状态机 vs LangChain Agent
  • 对话状态机：Java 开发者更好理解——状态（Start/AskDevice/AskScenario/Resolved）和状态转换规则
  • LangChain Agent：用 LLM 决定下一步行动（类比 AI Controller）。工具（Tool）= 可调用的函数（如查 FAQ、查数据库）。Agent 根据 LLM 的"思考"选择用哪个工具
  • 本阶段用 LangChain Agent 即可，不必手写状态机

Week 12：产品识别——自动判断用户问的是哪个产品

  1. 参考原文的"产品识别器"设计，在用户第一句话就判断产品类别：
    • 会议服务 / 可视电话 / 耳机 / 鼠标 / 会议大屏 / 通话服务 / 通用
  2. 用 LLM 分类 + 关键词规则双重校验：
    • LLM 分类：置信度高（>0.8）直接用
    • 置信度低 → 主动询问用户"您咨询的是哪款产品？"
  3. 产品确定后，加载对应知识库 collection，减少无关召回

Week 13：集成测试 + 对话质量评估

  1. 端到端测试：完整走一遍产品识别 → 意图识别 → 槽位填充 → RAG 检索 → 回答生成
  2. 建立评估集：
    • 收集 50~100 条真实用户问法（可以是模拟的，但要有代表性）
    • 人工标注：正确的意图、正确的产品、期望的回答
  3. 评估指标：
    指标含义目标
    意图准确率意图识别正确的比例> 90%
    产品识别率产品分类正确的比例> 95%
    回答准确率回答符合预期的比例> 80%
    对话完成率用户问题被解决的比例> 70%
  4. 整理测试结果，找出高频失败 case，分析原因（召回失败？意图误判？LLM 幻觉？）
做完后你要能回答：意图识别和实体抽取的区别是什么？槽位填充为什么重要？LangChain Agent 的 ReAct 模式是什么原理？
阶段产出
一个完整的多轮对话客服机器人：能自动判断产品线 → 判断意图 → 收集必要信息 → RAG 检索 → 生成回答。输出本阶段评估报告。
第 4 阶段 Agent 化升级 + 多产品线扩展

预计时间：第 14~16 周

你将理解什么是AI Agent，如何让 AI 主动调用工具、规划行动，并将其落地到 Zoo 的 6 大产品线。

核心技术点

LangChain Agent ReAct 模式 Tool Calling Function Calling 多产品线架构

任务清单

Week 14：深入理解 AI Agent

  1. 学习 AI Agent 的核心概念：
    • Agent = LLM + 工具 + 规划能力。能自主决定"我要做什么"
    • Tool = Agent 可以调用的外部能力（查数据库、调用 API、计算器、搜索）
    • ReAct（Reason + Act）= LLM 思考"我现在有什么信息，接下来该做什么"，然后执行 action，再观察结果
    • Tool Calling / Function Calling = LLM 输出结构化的工具调用指令，程序解析并执行
  2. 用 LangChain Agent 重构第 3 阶段的代码：
    • 把产品识别、意图识别、RAG 检索、人机转接判断都做成 Tool
    • Agent 根据对话上下文自主选择调用哪个 Tool
  3. 为 Agent 添加"自我检查"能力：生成回答后，让 Agent 判断置信度，低则转人工
Agent 实战架构（类比理解）
  • 把 Java 的思路翻译过来：Agent = 一个极度灵活的 Controller，不需要硬编码 if-else，它自己会推理下一步
  • Tool = 接口（Interface）。每个 Tool 有 name、description、function。Agent 根据 description 判断用哪个
  • Tool Calling = LLM 输出的 JSON，告诉程序"调用 tool_X，参数是 {a:1, b:2}"，程序解析后执行

Week 15：扩展到全部 6 条产品线

  1. 为剩余 4 条产品线准备知识库：
    产品线典型意图知识文档来源
    可视电话设备激活、固件升级、音视频配置产品手册 PDF
    耳机连接配对、音质问题、认证查询FAQ + 工单
    鼠标大屏控制、批注操作、配对问题快速入门指南
    会议大屏投屏、白板使用、无法入会运维文档
  2. 复用第 2 阶段的 RAG 流程，为每条产品线建立独立的向量 collection
  3. 复用第 3 阶段的意图分类体系，为每条产品线定义专属意图
  4. 添加产品切换能力：用户中途换话题（如"对了，耳机怎么连"），Agent 能动态切换知识库

Week 16：添加工具调用能力（Function Calling）

  1. 设计并实现几个实用 Tool：
    • QueryProductDoc：查 Zoo 产品文档（RAG）
    • QueryFAQ：查 FAQ 数据库（精确匹配）
    • CheckOrderStatus：查订单状态（模拟 API 调用，不接真实支付系统）
    • CreateTicket：创建人工工单（输出结构化工单信息，用于转接人工）
    • EscalateToHuman：判断是否需要转人工
  2. 用 OpenAI Function Calling（或 LangChain Agent 的 tool_call 机制）实现工具选择
  3. 处理工具调用失败的情况（超时、API 不可用），确保对话不中断
做完后你要能回答：AI Agent 和普通 Chain 的本质区别是什么？ReAct 循环是怎么工作的？Function Calling 和普通 Prompt 调用工具相比有什么优势？
阶段产出
一个 Agent 化的 Zoo 全产品线客服系统。能自动选择工具、自主决策下一步。可演示：用户问"耳机连接问题→排查→解决或转人工"全流程。
第 5 阶段 召回策略优化与调优

预计时间：第 17~18 周

这是提升系统质量的关键阶段。你将学会：如何量化评估召回质量、如何优化 chunk 策略、如何做 A/B 测试，以及如何持续迭代。

核心技术点

召回率评估 Chunk 策略调优 Hybrid Search 重排序 (ReRank) A/B 测试 Prompt 调优

任务清单

Week 17：召回质量分析与优化

  1. 建立召回评估集：
    • 准备 100 条测试问题，每条标注"期望召回的文档"
    • 计算召回率（Recall@K）：Top K 结果中包含正确答案的比例
  2. 分析召回失败的 case：
    • Embedding 模型不合适？换用 BGE、Sentence-BERT 等中文友好的 embedding 模型
    • Chunk 切分不合理？尝试不同 chunk_size（256/512/1024）和 overlap
    • 关键词丢失？引入 BM25 补充关键词召回（Hybrid Search）
    • 向量库参数问题？调整 FAISS 的 nprobe（搜索精度参数）
  3. 实现并对比以下召回策略：
    策略做法适用场景
    纯向量检索只靠 Embedding 相似度语义相近、表述多样的问题
    BM25关键词精确匹配含具体型号、错误码的问题
    Hybrid Search向量 + BM25 加权混合通用场景，推荐默认方案
    Rerank先召回 Top 20，再用 Cross-Encoder 重排召回量大、精确度要求高
  4. 用 LangChain 的 EnsembleRetriever 或手写加权逻辑实现 Hybrid Search
  5. 尝试 ReRank：用 BGE-Reranker 或 Cohere Rerank API 对召回结果重新排序
召回优化调参经验（实战技巧）
  • 先测 Recall@5（Top 5 召回准确率），如果低于 80%，说明 embedding 模型或 chunk 策略有问题
  • 中文场景推荐 embedding 模型：BGE-large-zh、m3e-large、text2vec-base-chinese
  • Hybrid Search 的权重比例（向量:BM25）可以从 0.7:0.3 开始调
  • Rerank 是召回的最后一道关卡，适合对精确度要求极高的场景（如医疗、金融客服）

Week 18：Prompt 调优 + A/B 测试 + 上线总结

  1. Prompt 调优（迭代式）：
    • 收集低质量回答的 case，分析 Prompt 哪里引导不足
    • 逐步优化 system prompt：添加回答格式要求、添加"拒答策略"、添加引用标注指令
    • 用 few-shot example（在 Prompt 中加示例）提升 LLM 对特定意图的回答质量
  2. A/B 测试设计（轻量版）：
    • 同一问题，对比两种召回策略的最终回答质量
    • 对比不同 Prompt 版本的效果
    • 记录每次实验的参数和结果，建立实验日志
  3. 系统总结（重要！做完后能讲出来）：
    • 整理项目架构图
    • 写出每个阶段的技术选型理由
    • 记录踩坑经历和解决方案
    • 整理一份"我学到了什么"的复盘文档
